1. Policy logic

The architecture of the prototype is described in more detail below, but the core of the application is the Gradated Access Service (GAS) which combines the following information into a decision about whether a request for a record should be allowed:

  • The user’s identity - an anonymous user, a logged-in member of TNA staff, or member of some other defined group

  • The user’s location - the external internet, an on site invigilation room, or other known location

  • Metadata of the record requested - title, description, opening date, foi exemption, etc

  • Policies defining access rules based on the above information

1.1. Request for an individual record

By default, a request for a record is denied. If we did not define any rules in the system, all requests would be rejected.

To enable some records to be accessible, we define a number of rule sets. Rule sets are groups of rules that apply to a specific combination of user and location, defined in a JSON format[1]. A rule sets defines all the rules that apply in order for a user in a particular location to see a record.

For example, we could define this rule set that applies to a journalist in the press room.

Access rules for a journalist in a press room
{
  "pressRoom": {
    "journalist": {
      "foi": "closed",
      "openingSoon": true
    }
  }
}

How we determine the identity and location of the user is beyond the scope for the Gradated Access prototype; we assume that there is is some authentication system or systems that can identify a user and their location - for example a journalist might log in with TNA provided credentials, and the press room would have a known address range.

This particular rule set contains two rules:

  • "foi": "closed"

  • "openingSoon": true

For be allowed to access a record, all of the rules in rule set must pass.

The records themselves don’t have the concept of 'opening soon'. The interpretation of the rules - what "openingSoon": true means, and whether a particular record is, indeed, opening soon - is the job of the rule implementation code.

The rule implementations are defined in the rules folder of the GAS. In this particular case, the openingSoon rule’s expected value is true, which the code interprets as a record where the openingDate property is at most 14 days from today. The exact value is hard-coded in the prototype but would be configurable in a real system. The openingSoon rule’s code, together with a brief commentary, is given in The openingSoon Rule Implementation.

This means that the complete definition a rule set is split between the policy - the JSON document that links sets of rules to user identity and location - and the implementation of the rules in code.

The rule implementation code interprets some rules as must rules and some as may rules. For example, "foi": "open" is interpreted strictly: a record must be open for it to be visible according to this rule. However, its reverse, "foi": "closed", is interpreted more flexibly: a record may be closed or open in order to be visible. This makes it easier for us to define rules that aren’t too restrictive.

1.2. Alternative rule sets

Take the example from above, which lets a journalist in the press room have access to records that are opening soon:

Access rules for a journalist in a press room
{
  "pressRoom": {
    "journalist": {
      "foi": "closed",
      "openingSoon": true
    }
  }
}

What if we also want to give journalists access to records that are in the cooling off period, that is those records which have just opened and are not yet generally available? One way the prototype handles this is to allow us to define multiple rule sets for each user/location combination.

Alternative rule sets, allowing access to documents about to open & to documents in their cool off period
{
  "pressRoom": {
    "journalist": [
      {
        "foi": "closed",
        "openingSoon": true
      },
      {
        "foi": "open",
        "coolingOff": false
      }
    ]
  }
}

For a record to be visible to this user, at least one of the rule sets must have all its rules pass. In other words, the rules within a set are combined with AND, and the the rule sets are combined together with OR.

1.3. Base policies

With the system described so far, defining a comprehensive collection of access rule sets quickly becomes exhausting. Let us say a journalist should have early access to some records if they are in the press room, but should have the same access as an anonymous user when outside TNA. Conversely, an unauthenticated anonymous user shouldn’t be able to see closed records, even if they are in the press room. A policy covering this scenario might look something like the following.

Exhaustive rule sets for two user identities and two locations
{
  "web": {
    "everyone": [
      {
        "copyright": "none",
        "foi": "open",
        "upsetting": false
      }
    ],
    "journalist": [
      {
        "copyright": "none",
        "foi": "open",
        "upsetting": false
      }
    ]
  },
  "pressRoom": {
    "everyone": [
      {
        "copyright": "none",
        "foi": "open",
        "upsetting": false
      }
    ],
    "journalist": [
      {
        "foi": "closed",
        "openingSoon": true
      }
    ]
  }
}

What we’ve described so far could be used to define any type of access, based on user identity and location. However, each combination of identity and location might require a rule set of its own, and it is extremely likely that many rule sets would be identical or largely identical. Even the simple example above almost immediately became repetitious - of its four rule sets, three are identical. Adding a new location or user identity would require a full suite of new rule sets. A change in TNA policy - adding or removing an access criteria, say - could potentially require every rule set to be changed. This would be unwieldy at best, and likely to be unworkable in practice.

To bring this approach back into the realm of the tractable, the GAS has a number of refinements to the approach described thus far. We tackle the problem of repetition through the use of base rule sets.

Base rule sets provide the defaults
{
  "base": {
    "base": [
      {
        "copyright": "none",
        "foi": "open",
        "upsetting": false
      }
    ]
  },
  "pressRoom": {
    "journalist": [
      {
        "foi": "closed",
        "openingSoon": true
      }
    ]
  }
}

The GAS builds a policy from the different rule sets starting the base rule set, then merging the identity- and location-specific rule set into it. If there is no specific rule set for an identity/location combination, then only the base rule set applies.

The base and specific rule sets are merged on a per-rule basis. A rule in the specific rule set replaces a rule with the same name in the base rule set. Any other rules, where there is no name match, are simply added into the merged set.

Applying that process here, the press room/journalist rule "foi": "closed" would replace "foi": "open" in the base rule set, and "openingSoon": true would be added. The resulting policy is as shown below.

Merged policy for journalist in press room
{
  "copyright": "none",
  "upsetting": false,
  "foi": "closed",
  "openingSoon": true
}

1.4. Location base policies

The base policy eliminates a great deal of potential duplication, but doesn’t eliminate it all. The base policy above gives access to open records - those which have no FOI exemption, no copyright restriction, and which are not considered upsetting.

Within TNA reading rooms, those restrictions can be loosened slightly and we are able to provide records which are covered by third party copyrights. To express this in our access policy would again put us in the position of repeated identical, or near identical, rule sets for each of our user identities.

Reading room rules repeated for each identity
{
  "base": {
    "base": [
      {
        "copyright": "none",
        "foi": "open",
        "upsetting": false
      }
    ]
  },
  "readingRoom": {
    "everyone": [
      {
        "copyright": "permit"
      }
    ],
    "journalist": [
      {
        "copyright": "permit"
      }
    ],
    ... all other identities
  },
  ... all other locations
}

Once again, we risk a combinatoric explosion of rule sets. Any progress we might have made seems illusory as we head back towards a maze of little rule sets all alike, and the corresponding maintenance quagmire.

It’s sometimes said that all problems in computer programming can be solved by adding another layer of indirection. We can apply that same principle here by introducing location base rule sets. These define default behaviour for a location, modifying or supplementing the base rule set, and potentially modified or supplemented by identity specific rule sets.

For the case under discussion - records with a copyright restriction are available in reading rooms - we can write that as shown below.

Reading room base rule set removes duplication
{
  "base": {
    "base": [
      {
        "copyright": "none",
        "foi": "open",
        "upsetting": false
      }
    ]
  },
  "readingRoom": {
    "base": [
      {
        "copyright": "permit"
      }
    ]
  },
  ... other locations base rule sets
}

Policies for a given user identity and location are synthesized in a similar way to that already described. The GAS starts with the base rule set, merges the location base rule set, finally merging that with the identity/location specific rule set. Rule sets are merged on a per-rule basis, with a rule in the more specific rule set replaces a rule with the same name. Any other rules, where there is no name match, are simply added into the merged set. It possible therefore, although unlikely, that a rule might be defined in the base rule set, modified by the location base rule set, and then modified again by an identity/location rule set.

Location base rule sets reduce repetition considerably. Our experiments with the prototype showed that specific policies become the exception, and generally occur for a single identity/location pair, such as the journalist/press room combination.

1.5. Rule Overrides

Sometimes, it’s useful to be able to override multiple rules at once. For example, a member of the FOI team who is in the TNA offices should be allowed to view all records, regardless of normal restrictions like closure or copyright.

We could achieve this by defining all the individual rules for the staff member. Without wishing to sound like a stuck record, or the 21st century equivalent thereof, we again risk both repetition and the maintenance burden of keeping them in step with the base definition.

Explicitly defining each rule for TNA staff
{
  "staffOffice": {
    "tnaStaff": [
      {
        "foi": "closed",
        "copyright": "permit",
        "upsetting": true
      }
    ]
  }
}

Instead, we define a new, more general, rule simply called allDocuments. The allDocuments implementation override all other rules, effectively removing them from the rule set.

Concise staff rule set
{
  "staffOffice": {
    "tnaStaff": [
      {
        "allDocuments": true
      }
    ]
  }
}

Overriding all other rules is a reasonably extreme case. In most cases where a rule overrides another, only a single named rule is overridden.

1.6. Policies for individual records

So far, we’ve been assuming that the rule sets are defined in a single file. In some ways, this is quite useful, because we can see all the rules in one place and, hopefully, understand the overall policy.

Imagine, though, that we want to add some very specific rules to a small group of records. Perhaps we want to provide access to a particular series for the duration of a public enquiry, or something of that nature.

We could support this by adding a new rule set, or series of rule sets, to our global policy. This is probably manageable, but mixing special cases and exceptions in with the general case serves largely to obscure.

The prototype supports adding extra rules at an individual record, subseries, or series record level, by defining an access.json rule alongside the record’s metadata. These extra rule sets are merged with the default rules before any of the base rule inheritance.

Appendix A: The openingSoon Rule Implementation

gas/lib/document/openingSoon.js
const { DateTime } = require('luxon')
const { dateStr } = require('../../util/rule-constants')

const OpeningSoon = 14

module.exports = {
  describe: value => { (1)
    return `The record ${value ? 'can be opening soon' : 'must be open'}`
  },
  evaluate: (metadata, value, now = DateTime.local()) => { (2)
    const openingSoon = openingDate(metadata)
      .diff(now, 'days')
      .toObject()
      .days

    if (openingSoon < 0)
      return true // already open

    return (openingSoon <= OpeningSoon) && value
  },
  reason: metadata => { (3)
    return `This record will be available for pre-opening viewing from ${dateStr(preOpenDate(metadata))}`
  },
  keyDate: metadata => [ (4)
    preOpenDate(metadata),
    openingDate(metadata)
  ],
  overrides: 'coolingOff' (5)
}

function openingDate(metadata) {
  return DateTime.fromISO(metadata.record.openingDate)
}

function preOpenDate(metadata) {
  return openingDate(metadata)
    .minus({days: OpeningSoon})
}

All rules follow this general pattern -

1 Human readable description of the rule.
2 The logic to evaluate whether the rule has passed or failed. For this openingSoon rule,

  • we calculate the number of days between the record’s opening date and now

  • if that number is negtive, the opening date is actually in the past - the record is already open and we can return true to indicate the rule has passed

  • otherwise, return true if the number days is less than our threshold value of 14

    All the rules in the prototype are similarly concise. Small targetted rules focussed on a single record attribute are easier to combine and reason about than longer, more complex rules looking at multiple aspects of the record.
3 In the event the rule has failed, a human readable explanation or mitigation that can be shown to a user.
4 Key dates are those dates on which the rule is likely to move from pass to fail or vice versa. In this particular case, those dates are 14 days before the record opening date, and the opening date itself.
5 The names of any other rules suppressed by this rule.

Appendix B: Deployment of the Prototype to AWS

The prototype consist of:

  • A frontend project, built in as a single-page app React

  • A "backend", which has multiple components:

    • The Gradated Access Service, a Node.js webapp

    • The Repo service, a separate Node.js webapp

    • The archive folder, a collection of documents and policy JSON files stored on disk

This is too complex to easily deploy to a typical prototype hosting service like Heroku, so we deployed the components to AWS as two separate services.

The frontend single-page app is deployed to AWS S3 as a static site.

All the backend services (including the documents in the file system) are bundled into a single Docker image, which is running in on AWS Fargate in ECS, using an application load balancer to handle incoming requests.

Within the Docker container, the Gradated Access Service handles incoming HTTP requests. It calls the Repo service over localhost, and the Repo service reads documents from the file system within the container.

This specific setup is not designed to be used for an app with sensitive data or with particular security considerations. It makes use of a publicly-readable S3 bucket and public subnets. This is only appropriate because this is a prototype that we are happy to share publicly. Do not copy this configuration if you are working with anything sensitive!

B.1. AWS configuration

All the services were configured manually in the AWS console. We considered writing a script using something like Terraform, but decided it wasn’t worth the time investment for a single prototype.

B.1.1. Initial DNS configuration

  • Set up DNS records

    • In AWS Route 53, create a hosted zone with the subdomain you want, e.g. da-access-prototype.nationalarchives.gov.uk

    • AWS will assign four name servers to the hosted zone. Give these name servers to the webmaster so that the subdomain is delegated to Route 53.

  • Create certificates

    • In AWS Certificate Manager, request a certificate for the subdomain

    • Choose DNS validation

    • Once you have confirmed the certificate request, click the domain name to expand the certificate details, and click "Create record in Route 53"

    • Repeat the certificate creation for the sub-subdomain that will be used for the API, e.g. api.da-access-prototype.nationalarchives.gov.uk

B.1.2. Frontend configuration

  • Create an S3 bucket

    • In the bucket properties, enable static website hosting

      • Set the index document to be index.html

    • Edit the bucket policy to allow all public read access

  • Deploy the code

    • Build the frontend project with npm build

    • Copy the generated build directory to S3 with the AWS CLI sync command

  • Set up CloudFront

    • In AWS CloudFront, create a distribution

    • Select the Web delivery method

    • Set the origin domain name to the S3 bucket

    • Do not set an origin path

    • Set the SSL certificate the custom certificate you created earlier for da-access-prototype.nationalarchives.gov.uk

  • Configure DNS for S3

    • In Route 53, open the hosted zone you created earlier

    • Create a Record Set

    • Leave the name blank (because you want to create a record at the top level of the subdomain)

    • Set the Type to "A - IPv4 address"

    • Set Alias to Yes

    • For the Alias Target, select the CloudFront distribution you created

    • Click Create

Once the DNS records have propagated, you should be able to visit the frontend at https://da-access-prototype.nationalarchives.gov.uk, but it will not be able to access any data because the backend has not yet been deployed.

B.1.3. Backend configuration

For all of these steps, make sure you are in the appropriate AWS region (e.g. London), not the default US East region.

  • Set up Docker image

    • Build the Docker image using the prototype’s Dockerfile

    • Create a Docker image repository in ECR

    • Push the image to ECR using the commands shown when you click "View push commands"

  • Create the load balancer

    • In the EC2 section of the AWS console, create a new load balancer

    • Select application load balancer (HTTP/HTTPS)

    • Set the protocol to be HTTP (port 80) for now - we’ll come back and and configure HTTPS in a later step

    • Select the default VPC, and all of its availability zones

    • When prompted for a security group, create a new one

    • When prompted for a target group, create a new one

    • Skip the Register Targets step - we haven’t created a target to point the load balancer at yet

    • When your new load balancer has been created, open it and delete the listener that was created automatically

  • Configure image deployment

    • Create a new task definition

      • Set the launch type to Fargate

      • Set the task memory and CPU to the minimum - this has been fine for running the prototype so far

      • Add a container definition

        • Set the image name to the image URI listed in the ECR repo - choose the one with the latest tag

        • Set the port mapping to 3005 (TCP)

    • Create a new cluster in AWS ECS

      • Use the "Networking only" template

    • Create a new service in the cluster you just created

      • Set the launch type to Fargate

      • Choose the task definition you created earlier

      • Set the number of tasks to 1

      • Set the cluster VPC to the default VPC

      • Choose all of the VPC public subnets

      • Set the load balancer type to Application Load Balancer

      • Select the ALB that you created earlier

      • Select the container you configured earlier, and click "Add to load balancer"

      • Set the listener port to 80

      • Leave the rest of the settings on that page as the defaults

      • For service auto-scaling, select "Do not adjust"

      • Click through the rest of the service setup

  • Check the security group configuration

    • The security group of the load balancer should allow HTTP and HTTPS traffic from any IP

    • The security group of the ECS service should allow traffic to port 3005 from the ALB’s security group

  • Monitor the Tasks and Events tabs of the ECS service to make sure the container has started up correctly

At this point, you should be able to visit the URL of the load balancer over HTTP (see the DNS name in the ALB Description tab) and see the prototype.

  • Configure HTTPS access to the load balancer

    • Open the ALB you created earlier

    • Add a new listener

    • Set the protocol to HTTPS (port 443)

    • Set the default action to forward to a target group, and choose the API target group you created when you set up the ECS service

    • Select the ACM certificate for the API that you created earlier

  • Configure DNS

    • In Route 53, create an A record for api.da-access-prototype.nationalarchives.gov.uk like you did for the frontend

    • Set the target to the ALB

Once the DNS records have propagated, the frontend should show data from the backend, such as the list of records in the "What?" dropdown.
